# 阿德小厨房 · 跨设备项目记忆

> 本文件是 Mac、Windows 与其他开发端之间共享的长期项目记忆。任何接手本仓库的 Codex/AI 助手，都应在修改代码前完整阅读本文件和 `README.md`，并以仓库中的最新代码为事实依据。

## 1. 项目主人与目标

- 项目主人：adechokwok（阿德）。
- 产品名称：阿德小厨房。
- GitHub 公开仓库：`adechokwok/ade-private-kitchen`。
- 这是家庭主厨自己的私人点菜与家宴管理软件，不是面向大众的平台。
- 朋友通过手机网页浏览菜品、选择菜品并提交饭局；主厨在工作台接单、管理菜单、查看食材、生成采购清单和编排家宴菜单。
- 长期运行环境是极空间 Z4Pro NAS，程序通过 Docker 运行，菜谱、订单和照片保存在 NAS 硬盘。

## 2. 产品体验原则

- 朋友点菜端应简单、温暖、俏皮，减少专业参数和不必要的输入。
- 主厨可持久化切换今日营业/休息状态，朋友端用绿灯或红灯同步显示。
- 品牌文字使用“阿德小厨房”，品牌标记使用“德”。
- 主厨工作台可以更完整，但重要操作应使用普通中文，避免让家庭用户理解技术术语。
- 菜品类型必须可自定义，不限制为“家常热炒”等固定类别。
- 客人端不展示制作时间、几人份、口味等主厨内部信息。
- 菜谱录入应支持文字、截图/照片识别、本地上传菜品照片；自动找图只能使用来源和授权清晰的图片。
- 菜品介绍保持 22–42 个中文字符、凝练且有具体画面；点菜 slogan 保持 7–14 个中文字符、俏皮并贴合这道菜独有的口感或吃法，避免广告腔和通用 AI 套话。
- 家宴菜单提供 10 套与场合一致的独立模板；背景只负责边缘氛围，菜名和正文始终由 HTML/CSS 排版并保持高对比。
- 移动端体验优先，同时保持桌面端和 Windows 浏览器可用。

## 3. 当前主要功能

- 朋友点菜首页、菜品分类与选择；手机端使用左侧固定分类、右侧连续菜品列表，顶部用独立横向卡片展示“阿德推荐”，电脑端也有独立主厨推荐区，分类会随右侧滚动自动高亮。
- 私人邀请链接、推荐菜和点菜提交。
- 订单状态与主厨进度反馈。
- 主厨工作台按“接单 → 订单和采购 → 制作 → 开饭”组织；接单页汇总饭局并合并宴席编排器，完成订单自动进入归档。
- 制作执行台自动同步进行中订单、用餐信息、忌口、按人数换算的用料与完整步骤，并提供合并备菜、按开饭时间倒排、单菜计时器、库存不足提醒和本机步骤勾选。
- 主厨密码登录与安全会话。
- 菜品新增、编辑、上下架和自定义分类；新菜优先从已有类型中选择，只有选择“自定义新类型”后才能输入新类型。
- 每个分类可保存自定义 Emoji，并可新增、改名、排序和删除；已有与新建分类都能移动到任意位置，删除分类后其余顺序保持稳定，其中菜品统一移入“未分类”，不得连同菜谱删除。
- 主厨端完整菜谱库使用带滚动条的连续瀑布流，不使用分页或“继续加载”按钮；支持多选菜品后批量加入指定大类，菜谱卡片不提供复制操作。
- 已归档菜品仍可直接编辑、恢复或永久删除，不因归档失去管理入口。
- 菜品封面、步骤图片与家宴照片的本地存储。
- 菜品封面支持本地智能主体取景、拖动与缩放微调；只保存取景参数，不重复生成或破坏原图。
- 智能菜谱录入；通过 DashScope OpenAI 兼容接口调用通义千问 Qwen3-VL-Plus，支持截图方向校正、图片理解和完整步骤提取；不配置时仍可使用文字解析。
- 菜谱识别会同时生成可编辑的菜品介绍和点菜 slogan；菜单编辑可分别调用千问重新生成，介绍强调具体口感与香气，slogan 使用熟人间俏皮自然的口吻。
- 主厨端支持批量选择 JSON 菜谱文件，先预览新增与更新数量，再自动备份 SQLite 并用事务按菜名合并；同名菜保留原有照片和菜单状态，导入失败整批回滚。
- 食材拆解、采购清单、家中库存与采购勾选状态。
- 家宴菜单编排、模板切换，并可把正式菜单卡片直接导出为 PNG、JPG 或 PDF；内置温馨家宴、二人世界、Fine Dining、新春团圆、中秋雅宴、生日烛光、乔迁暖居、夏日晚风、冬日圣诞和周末早午餐 10 套独立主题背景。
- 饭后家宴日记。
- SQLite 数据库、健康检查和每日备份。

## 4. 技术与数据架构

- Web：Next.js 16、React 19、TypeScript。
- 运行时：Node.js 22.13 或以上，pnpm 11.9.0。
- 数据库：NAS 本地 SQLite，使用 `better-sqlite3` 和 Drizzle ORM。
- 图片：NAS 本地文件系统，不依赖 Cloudflare R2。
- 主厨认证：应用自己的密码和 HMAC 签名会话，不依赖第三方登录。
- 正式部署：Docker `linux/amd64`，对应极空间 Z4Pro。
- 正式镜像：`ghcr.io/adechokwok/ade-private-kitchen:latest`。
- 正式 Compose：`compose.nas.yaml`。
- 本地源码构建 Compose：`compose.yaml`。

持久数据必须始终保留在宿主机挂载目录：

```text
/data/      SQLite、会话密钥、菜品与家宴照片
/backups/   数据库历史备份与最新照片镜像
```

禁止把用户数据写进 Docker 镜像或容器临时层。升级和重新创建容器时，不得删除这两个目录。

## 5. 固定发布与更新流程

项目主人已确认使用以下主线流程：

```text
提出需求 → 实现并检查 → 提交到 GitHub main
→ GitHub Actions 自动测试 → 构建 GHCR 镜像
→ 极空间人工确认拉取 latest → 重新部署容器
```

- GitHub 工作流：`.github/workflows/publish-nas-image.yml`。
- Windows 端发布优先使用已限定到本仓库的 ChatGPT Codex GitHub Connector：在远端创建 `agent/...` 分支，组成原子提交并打开草稿 PR，确认后合并到 `main`，再检查 Actions 与 GHCR。若本机 `gh auth`、Git Credential Manager 或 OAuth 令牌接口超时，不再要求项目主人反复登录，直接复用这条已授权连接器路径。
- 通过 Connector 上传大文件时，必须把本地文件按小块读取后在内存中拼成完整 Base64，再创建 Git blob；不得直接使用可能被输出上限截断的整文件命令结果。提交后必须对远端文件大小或 SHA-256 做完整性核验，再允许合并和发布镜像。
- 每次成功发布同时生成 `latest` 和 `sha-完整提交编号` 两种标签。
- NAS 默认保留人工确认更新，避免饭局期间被自动重启。
- 程序更新前确认最近备份正常；出现问题时使用上一个 `sha-...` 镜像回退。
- 除非项目主人明确改变策略，否则完成功能并通过检查后直接同步到 `main`。

## 6. 开发和验收规则

开始工作前：

1. 阅读本文件、`README.md` 和相关代码。
2. 检查当前分支、远端状态和未提交修改。
3. Windows 与 Mac 两端都先同步 GitHub 最新版本；不得覆盖另一端未合并的工作。
4. 以数据库和现有 API 的实际实现为准，不凭聊天记忆猜测。

完成工作前：

1. 运行 `pnpm test`。
2. 运行 `pnpm lint`；已有图片优化提示可以记录，但不得新增错误。
3. 涉及 Docker 时确认 `linux/amd64`、健康检查、数据挂载和备份服务仍然有效。
4. 涉及数据库结构时优先兼容旧数据，并说明迁移与回退方式。
5. 提交范围应只包含本次需求，不覆盖用户或另一设备的无关修改。
6. 推送后检查“测试并发布 NAS 镜像”是否成功；失败时读取日志并修复，不把失败流程留给用户。

## 7. 安全边界

- 不得在代码、本文档、提交记录或聊天中保存密码、GitHub Token、API Key、NAS 公网地址等秘密。
- `.env` 只保存在对应设备或 NAS；仓库只保留 `.env.example` 与 `nas.env.example`。
- GHCR Package 设为公开时 NAS 可匿名拉取；若临时改回私有，只使用具备 `read:packages` 权限的凭证。
- 删除数据库、照片、备份、Docker 卷或迁移旧数据前，必须获得项目主人明确同意。
- 公开互联网访问需要 HTTPS、可靠域名/隧道和访问控制；不能直接把 NAS 管理端口暴露到公网。

## 8. 跨端协作方法

- Mac 和 Windows 必须使用同一个 GitHub 仓库，不依赖某一台电脑的聊天记录。

- 如果本文件与最新代码冲突，以最新代码和项目主人的当前明确要求为准，并随后修正本文件。
